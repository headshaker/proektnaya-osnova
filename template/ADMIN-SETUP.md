# Настройка проекта для технического специалиста

[← Главная](README.md) | [Старт руководителя](START-HERE.md) | [Подключение ИИ](AI-CONNECTIONS.md) | [Обновление основы](MIGRATIONS.md)

> [!warning] Техническая инструкция
> Эта страница предназначена для специалиста, который настраивает окружение, Git и агентный доступ. В обычной работе руководителю эти действия не требуются.

## Ожидаемый результат

После настройки руководитель должен иметь возможность:

1. открыть [пульт руководителя](HOME.md);
2. ставить задачи ИИ обычным языком;
3. добавлять материалы в `_attachments`;
4. читать управленческий итог;
5. согласовывать публикацию без работы в терминале.

## Самый простой запуск

### Windows

Дважды щёлкните [`START-PROJECT.cmd`](./START-PROJECT.cmd). В официальном выпускном архиве Electron и PowerShell уже встроены, поэтому обычному пользователю не требуется устанавливать их или запускать сценарии вручную. Визуальный мастер:

1. проведёт по шести коротким экранам без командной строки;
2. спросит название и предложит техническое имя;
3. поможет выбрать строгость управления и способ организации работ;
4. спросит о рабочей системе, чувствительности данных и допустимых отклонениях;
5. отдельно выполнит предварительную проверку, не изменяя файлы;
6. только после подтверждения настроит проект и выполнит встроенные проверки;
7. позволит выбрать несколько нейросетей и Obsidian, проверит их наличие и покажет инструкции для отсутствующих программ;
8. создаст единый файл [`AI-TOOLS.json`](./AI-TOOLS.json), адаптеры инструкций и переносимую настройку Obsidian без секретов;
9. если проект уже является отдельным GitHub-репозиторием, автоматически защитит ветку `main` и добавит обязательную проверку **«Одна согласованная версия проекта»**;
10. перечислит решения и подключения, которые ещё нужно завершить.
11. включит на каждом настроенном компьютере Git-хуки и фоновую проверку `main`, которая обновляет папку и общий контекст выбранных нейросетей без перезаписи локальной работы.

Автоматическая защита использует текущий вход в GitHub CLI (`gh`) и не сохраняет токен в проекте. У запускающего должны быть административные права на репозиторий. Если GitHub ещё не подключён или прав недостаточно, остальные файлы проекта будут настроены, а мастер явно отметит защиту как незавершённую.

Проверка программ не запускает сетевые установщики. Если программы нет, пользователь открывает её официальную страницу кнопкой в мастере, устанавливает обычным для своей системы способом и повторяет проверку. Мастер не предлагает руководителю копировать команды PowerShell. Пароли и API-ключи вводятся только в интерфейсе соответствующего поставщика или безопасном хранилище окружения.

### Windows, macOS или Linux через терминал

    pwsh ./scripts/setup-project.ps1

Для автоматизации сначала можно получить только план:

    pwsh ./scripts/setup-project.ps1 -Title "Название проекта" -NonInteractive

Чтобы применить проверенный план, добавьте `-Apply`.

В автоматизированном развёртывании можно потребовать завершённую защиту GitHub и считать её отсутствие ошибкой:

    pwsh ./scripts/setup-project.ps1 -Title "Название проекта" -NonInteractive -Apply -GitHubProtectionMode required

## Требования

- для официального выпускного архива Windows отдельный PowerShell и Electron не нужны — оба компонента встроены;
- PowerShell 7 (`pwsh`) требуется только при техническом запуске из исходного репозитория или в macOS/Linux;
- при запуске прямо из исходного репозитория без встроенного runtime — PowerShell 7, Node.js 22.12+ и интернет для однократной установки точной версии Electron; такой запуск предназначен для технического специалиста;
- Git и отдельный репозиторий проекта;
- GitHub CLI (`gh`) с административным доступом — для автоматической защиты `main`;
- агентная среда с доступом только к этому репозиторию;
- Obsidian — по желанию, для удобного просмотра;
- Python 3.10+ и Microsoft MarkItDown — только для автоматической обработки офисных файлов и PDF.

## Ручная настройка как запасной вариант

Следующие шаги нужны, только если мастер недоступен или требуется нестандартная автоматизация.

### 1. Создайте проект

Скопируйте содержимое папки `template` в корень отдельного репозитория и выполните:

    pwsh ./scripts/init-project.ps1 -Title "Название проекта" -Slug "project-name"

`Slug` — короткое техническое имя строчными латинскими буквами, цифрами и дефисами, например `warehouse-modernization`.

### 2. Настройте управление

Откройте [`PROJECT-CONFIG.json`](./PROJECT-CONFIG.json) и задайте параметры проекта.

| Поле | Что означает | Частый выбор |
|---|---|---|
| `managementProfile` | строгость управления | `light`, `standard` или `regulated` |
| `deliveryApproach` | способ организации работ | `predictive`, `incremental`, `adaptive`, `flow` или `hybrid` |
| `workSystem` | где команда ведёт задачи | Jira, GitHub Issues, Linear или другая система |
| `reviewCadence` | как часто проводятся обзоры | обычно еженедельно |
| `tolerances` | допустимые отклонения | определяются владельцем проекта |
| `dataClassification` | чувствительность данных | обязательно согласовать до подключения внешних сервисов |
| `aiGovernanceLevel` | уровень контроля работы ИИ | `standard` либо более строгий режим |

Не угадывайте бюджет, допуски, классификацию данных и обязательства. Получите решение владельца.

### 3. Подключите ИИ

Используйте [инструкцию по подключению](AI-CONNECTIONS.md). Минимальные правила:

Повторно выбрать или проверить инструменты после первоначальной настройки можно командой:

    pwsh ./scripts/configure-project-tools.ps1 -AiToolsCsv "chatgpt,claude,gemini,qwen,deepseek,grok" -ObsidianMode enabled -Apply

- предоставляйте доступ только к репозиторию проекта;
- начинайте с прав чтения и записи файлов;
- не храните токены и пароли в репозитории;
- публикацию и слияние оставляйте за человеком;
- существенные изменения готовьте в отдельной ветке и на странице согласования.

Если подключено несколько нейросетей, выдайте каждой отдельный клон или рабочее дерево и включите [протокол координации](AI-COORDINATION.md). Мастер автоматически создаёт GitHub Ruleset **«Проектная основа: единая версия»** для ветки `main`. Он требует запрос изменений, свежую основу и прохождение проверки **«Одна согласованная версия проекта»**, а также блокирует удаление и принудительную перезапись `main`.

Если мастер сообщил, что защита не завершена, войдите в `gh` с учётной записью администратора репозитория и повторите только безопасный шаг настройки:

    gh auth login
    pwsh ./scripts/configure-github-protection.ps1 -Apply

Повторный запуск не создаёт дубликат: существующий Ruleset с этим служебным названием обновляется. Результат записывается локально в `.project/github-protection.json`; секреты туда не попадают.

### 4. Настройте просмотр

При использовании Obsidian:

1. установите приложение с официального сайта;
2. выберите **Open folder as vault**;
3. укажите корневую папку проекта;
4. закрепите [`HOME.md`](./HOME.md), [`STATUS.md`](./STATUS.md) и [`HANDOFF.md`](./HANDOFF.md).

Подробности находятся в [инструкции Obsidian](OBSIDIAN.md).

### 5. Проверьте готовность

Выполните:

    pwsh ./scripts/build-status.ps1
    pwsh ./scripts/check-project-health.ps1
    pwsh ./scripts/build-project-dossier.ps1
    pwsh ./scripts/build-project-dossier.ps1 --check
    pwsh ./scripts/validate-registries.ps1
    pwsh ./scripts/validate-vault.ps1

Исправьте ошибки, а предупреждения о цели, выгодах, рабочей системе, классификации данных и допусках передайте владельцу как вопросы, а не заполняйте предположениями.

### 6. Передайте проект руководителю

Покажите руководителю только:

- [пульт руководителя](HOME.md);
- [первую инструкцию](START-HERE.md);
- куда добавлять материалы;
- где читать итог ИИ;
- как разрешить публикацию или запросить доработку.

Терминал, команды проверки и структура служебных файлов не должны становиться обязательной частью ежедневной работы.
