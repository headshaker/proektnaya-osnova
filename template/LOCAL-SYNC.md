# Автоматическое обновление локальной папки

[← Пульт руководителя](HOME.md) | [Подключение ИИ](AI-CONNECTIONS.md) | [Совместная работа нейросетей](AI-COORDINATION.md)

## Что получает руководитель и участник

После однократного запуска [`START-PROJECT.cmd`](./START-PROJECT.cmd) на каждом компьютере система:

1. в Windows без видимого окна проверяет общую ветку `main` каждые пять минут и при открытии проекта; в macOS и Linux — при открытии и Git-операциях;
2. получает только безопасное обновление вперёд без переписывания истории;
3. после изменения Git автоматически пересобирает локальный пакет контекста всех выбранных нейросетей;
4. сохраняет результат последней проверки в `.project/local-sync-status.json`, а краткую историю работы и ошибок — в `.project/local-sync.log`;
5. при ручном `pull`, переключении ветки или завершении `rebase` также обновляет контекст через локальные Git-хуки.

GitHub не может сам изменять папки на закрытых компьютерах. Поэтому каждый руководитель и участник один раз запускает проект локально. В Windows после этого скрытая фоновая задача действует от имени пользователя, не требует административных прав и использует внутренний механизм официального архива. В macOS и Linux обновление срабатывает при запуске и Git-операциях; постоянный интервал при необходимости настраивает технический специалист системным планировщиком.

## Как выключить или снова включить обновление

Повторно запустите [`START-PROJECT.cmd`](./START-PROJECT.cmd). В окне настроенного проекта можно:

- выключить фоновое обновление только на этом компьютере;
- снова включить его без изменения настроек коллег;
- открыть локальный журнал работы;
- перейти в пульт руководителя.

Отключение удаляет фоновую задачу и локальные Git-хуки этого компьютера. Общая политика проекта в [`LOCAL-SYNC.json`](./LOCAL-SYNC.json) при этом не меняется.

## Что система никогда не делает автоматически

- не удаляет и не сбрасывает локальные изменения;
- не использует принудительный push или принудительное обновление;
- не объединяет разошедшиеся версии;
- не переносит ветку работающей нейросети поверх нового `main`;
- не публикует файлы и не сохраняет секреты.

Если в `main` есть локальные изменения, обновление останавливается. Если открыта ветка нейросети и общая версия изменилась, нейросеть получает статус `agent-branch-stale`, прекращает содержательную работу и выполняет осмысленную синхронизацию по [`AI-COORDINATION.md`](./AI-COORDINATION.md).

## Как обновляется контекст нейросетей

Общий локальный пакет находится в `.project/context/ai-package.md`. Его отпечаток, редакция Git, профиль и выбранные инструменты записываются в `.project/context/local-context-state.json`.

Перед новой задачей нейросеть обязана:

1. запустить безопасную проверку актуальности;
2. остановиться при статусе устаревшей ветки;
3. прочитать свежий пакет и только после этого продолжить работу.

Уже открытый внешний чат нельзя технически переписать задним числом. Поэтому активная нейросеть повторно проверяет редакцию перед каждой новой содержательной задачей, а новая сессия всегда начинает со свежего пакета.

## Проверка техническим специалистом

Разовая проверка:

    pwsh ./scripts/sync-project.ps1

Повторное включение фоновой задачи и Git-хуков:

    pwsh ./scripts/install-local-sync.ps1 -Apply

Обычному пользователю команды не нужны: состояние, журнал и переключатель доступны при повторном запуске `START-PROJECT`. В [`LOCAL-SYNC.json`](./LOCAL-SYNC.json) технический специалист может изменить общий интервал от 1 до 1440 минут или отключить механизм для всего проекта. Это политика проекта, поэтому её изменение проходит обычную проверку и публикацию.
