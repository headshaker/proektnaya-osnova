# Обновление проектной основы

[← Главная](README.md) | [Ежедневная работа](DAILY-WORK.md) | [История проекта](CHANGELOG.md)

> [!warning] Техническая инструкция
> Обновление версии выполняет специалист или агент с доступом к Git и терминалу. Руководителю достаточно получить план изменений, сведения о конфликтах и итог проверки простым языком.

Обновление запускается из распакованного архива новой версии, а изменяемый проект передаётся через `-ProjectPath`. Канонические материалы проекта — бриф, решения, вопросы, источники, профильные документы и передача контекста — автоматически не заменяются.

## Подготовка

1. Зафиксируйте текущие изменения проекта в Git.
2. Скачайте ZIP и SHA-256 новой версии и проверьте контрольную сумму.
3. Распакуйте выпуск отдельно от обновляемого проекта.
4. Сначала постройте план без изменения файлов:

       pwsh ./scripts/update-project.ps1 -ProjectPath "../my-project"

5. Изучите операции и конфликты. Затем примените согласованный план:

       pwsh ./scripts/update-project.ps1 -ProjectPath "../my-project" -Apply

## Гарантии

- управляемый файл заменяется без дополнительного разрешения только тогда, когда его SHA-256 совпадает с официальной версией исходного выпуска;
- изменённый управляемый файл создаёт конфликт; `-ForceManagedFiles` допустим только после ручного сравнения;
- пользовательские и канонические документы не заменяются новыми файлами шаблона;
- перед записью создаётся резервная копия в `.project/backups`;
- при ошибке проверки применённые операции откатываются;
- [`TEMPLATE-VERSION`](./TEMPLATE-VERSION) изменяется только в рамках применяемой транзакции.

## Старые проекты без маркера версии

Версии `0.1.0` и `0.1.1` могли не содержать [`TEMPLATE-VERSION`](./TEMPLATE-VERSION). Для них версия не угадывается:

    pwsh ./scripts/update-project.ps1 -ProjectPath "../old-project" -FromVersion 0.1.1

Указывайте `-FromVersion` только после сверки происхождения проекта. Неподдерживаемая или противоречащая маркеру версия отклоняется.

## Совместимость реестров

Файл [`REGISTRY-SCHEMA.json`](./REGISTRY-SCHEMA.json) описывает поддерживаемые строки решений, допущений, вопросов, источников и элементов управления. Проверка принимает исторические ID с любым количеством цифр (`D-1` и [`D-001`](./DECISIONS.md#d-001)) и форматы таблиц версий `0.1.x–0.9.x`, но отклоняет повторяющиеся ID и неизвестное число колонок.

    pwsh ./scripts/validate-registries.ps1

### Почему [`AGENTS.md`](./AGENTS.md) не заменяется

Команда могла адаптировать [`AGENTS.md`](./AGENTS.md), [`OBSIDIAN.md`](./OBSIDIAN.md), [`DAILY-WORK.md`](./DAILY-WORK.md) и другие инструкции под свой проект. Миграция сохраняет эти файлы. Новая модель подключается через добавочный [`AGENTS.override.md`](./AGENTS.override.md) и отдельные руководства.

После обновления:

1. сравните новые документы с собственными правилами проекта;
2. при необходимости вручную перенесите полезные разделы из новой редакции шаблона;
3. подключите агентный инструмент по [`AI-CONNECTIONS.md`](./AI-CONNECTIONS.md);
4. попросите агента провести первичную диагностику по [`AI-OPERATING-MODEL.md`](./AI-OPERATING-MODEL.md);
5. проверьте результат через отдельную ветку и draft pull request.

Если в проекте уже существует файл с тем же именем, additive-файл не заменяется автоматически. Сравнение и объединение выполняются вручную.

## Переход на 0.8.0

Версия `0.8.0` добавляет замкнутый управленческий цикл поверх существующей базы знаний:

- [`PROJECT-CONFIG.json`](./PROJECT-CONFIG.json) задаёт профиль управления, подход к поставке, внешнюю рабочую систему, ритмы обзора и допуски;
- [`OUTCOMES.md`](./OUTCOMES.md) связывает выгоды с владельцами, метриками, целями и доказательствами;
- [`CONTROLS.md`](./CONTROLS.md) объединяет риски, проблемы, зависимости и запросы на изменение;
- [`STATUS.md`](./STATUS.md) строится из канонических данных командой `pwsh ./scripts/build-status.ps1`;
- `pwsh ./scripts/check-project-health.ps1` проверяет конфигурацию, сроки, просрочки и обязательные элементы управления;
- [`AI-GOVERNANCE.md`](./AI-GOVERNANCE.md) определяет уровни риска, требования к доказательствам и человеческому подтверждению;
- [`.github/workflows/project-health.yml`](.github/workflows/project-health.yml) выполняет проверки статуса и здоровья при изменениях и по расписанию.

Все эти файлы добавляются безопасно: существующий файл с совпадающим именем не заменяется. Миграция также расширяет управляемую схему реестров идентификаторами `B-xxx`, `R-xxx`, `I-xxx`, `X-xxx`, `C-xxx` и `G-xxx`.

После обновления:

1. заполните [`PROJECT-CONFIG.json`](./PROJECT-CONFIG.json), особенно `workSystem`, `dataClassification` и применимые допуски;
2. внесите минимум одну проверяемую выгоду в [`OUTCOMES.md`](./OUTCOMES.md);
3. объедините полезные изменения из новой редакции [`AGENTS.md`](./AGENTS.md), [`CONTEXT-PROFILES.json`](./CONTEXT-PROFILES.json), [`scripts/build-context.ps1`](./scripts/build-context.ps1), [`scripts/project-dossier.manifest.json`](./scripts/project-dossier.manifest.json), [`docs/03-delivery.md`](./docs/03-delivery.md) и [`docs/04-governance-risks.md`](./docs/04-governance-risks.md), если они уже были адаптированы в проекте;
4. выполните `pwsh ./scripts/build-status.ps1`, затем `pwsh ./scripts/check-project-health.ps1`;
5. проверьте diff и зафиксируйте осознанные изменения отдельным коммитом.

Рабочие задачи по-прежнему остаются в Jira, GitHub Issues или другой выбранной системе. В репозитории хранятся только ссылки, управленческие условия, результаты, доказательства и решения.

1. проверить, что новые файлы добавились без конфликта;

3. продолжить обычную работу с [`HOME.md`](./HOME.md).

    pwsh ./scripts/build-context.ps1 -Profile standard -Check
    pwsh ./scripts/check-context-health.ps1 -UpdateBaseline -Check

## Переход на 0.10.1

Версия `0.10.1` автоматизирует защиту единой версии проекта на GitHub:

- [`scripts/configure-github-protection.ps1`](./scripts/configure-github-protection.ps1) создаёт или обновляет Ruleset для `main`;

Для применения Ruleset нужны административные права на репозиторий. Если их пока нет, завершите остальные шаги мастера, получите права и повторите команду из отчёта настройки. Токены в файлы проекта не записываются.

- русские пояснения для ChatGPT / Codex, Claude Code, Qwen Code и других инструментов отображаются без повреждения;
- запасной текстовый мастер больше не читает неинициализированную `$LASTEXITCODE` после запуска PowerShell-сценария.

Обновление заменит только неизменённые официальные файлы запуска и настройки `0.13.0`. Если команда меняла эти файлы, миграция остановится и предложит сравнение вместо скрытой перезаписи.

## Переход на 0.13.2

Версия `0.13.2` устраняет оставшиеся зависимости первого запуска от поведения Windows:

- результат проверки нейросетей передаётся как ASCII-безопасный JSON с Unicode-последовательностями, поэтому кодовые страницы `866`, `1251` и UTF-8 дают одинаковый текст;
- встроенный Electron-мастер запускается как отдельный отслеживаемый процесс, а код завершения берётся из него напрямую;
- повторный запуск больше не читает неинициализированную `$LASTEXITCODE` после открытия графического окна.

Обновление заменит только неизменённые официальные файлы `0.13.1`. Пользовательские изменения по-прежнему не перезаписываются автоматически.

## Обновление до 0.14.0

Версия `0.14.0` добавляет в визуальный мастер краткое обучение руководителя:

- новый шестой шаг объясняет ежедневный и еженедельный рабочий цикл без Git, терминала и ручного редактирования файлов;
- памятка разделяет ответственность руководителя, команды и ИИ и напоминает правила параллельной работы нескольких нейросетей;
- кнопки открывают существующие локальные руководства [`START-HERE.md`](./START-HERE.md), [`PROMPTING-GUIDE.md`](./PROMPTING-GUIDE.md) и [`TEAM-INPUT.md`](./TEAM-INPUT.md);
- список доступных из интерфейса документов зафиксирован в основном процессе Electron, поэтому произвольный путь открыть нельзя.

Обновление заменит только неизменённые официальные файлы мастера `0.13.2`. Уже настроенный проект повторно запускать не нужно: новые рекомендации остаются доступны в перечисленных руководствах и на [`HOME.md`](./HOME.md).

## Обновление до 0.14.1

Версия `0.14.1` делает первый запуск Windows автономным и понятным нетехническому пользователю:

- официальный архив включает Electron и PowerShell, поэтому устанавливать `pwsh` или вручную запускать сценарии не нужно;
- ярлык сначала открывает визуальный мастер и не переводит пользователя в текстовый режим при ошибке;
- команды установки заменены кнопками официальных страниц, а диагностический журнал отделён от действий руководителя;
- существующие проекты получают исправление только для неизменённых официальных файлов, пользовательские изменения остаются защищены.

## Восстановление

После успешного обновления резервная копия сохраняется до ручного удаления. Отчёт `update-report.json` содержит исходную и целевую версии, перечень операций и результат. При автоматическом откате изучите отчёт и устраните причину до повторного запуска.
