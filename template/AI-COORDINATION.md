# Совместная работа нескольких нейросетей

Этот протокол позволяет руководителю одновременно поручать работу Codex, Claude, Gemini, Copilot и другим нейросетям, сохраняя **одну каноническую версию проекта**.

## Коротко для руководителя

1. Каждой нейросети дайте отдельную задачу и не открывайте двум исполнителям одну и ту же рабочую папку.
2. В начале задания скажите: «Следуй [`AI-COORDINATION.md`](./AI-COORDINATION.md), зарегистрируй работу и не выходи за её границы».
3. Разрешайте объединение изменений по одному. После каждого объединения остальные нейросети должны обновить свою работу.
4. Если автоматическая проверка сообщает об устаревшей основе или выходе за границы задачи, не обходите её: поручите нейросети синхронизацию.

Нейросети могут одновременно анализировать разные вопросы. Канонические изменения объединяются последовательно — это цена сохранения одной непротиворечивой версии.

## Вход от команды без редактирования файлов

Команда проекта не меняет Markdown, JSON или другие файлы репозитория. Она только отвечает на открытые вопросы, добавляет ссылки на источники и прикладывает материалы через форму **«Ответ, источник или файл от команды»** в GitHub либо передаёт их в чате с ИИ.

При каждом запуске ИИ сначала проверяет очередь с меткой `team-input:queued`, берёт одно сообщение, проверяет источник и вложения как недоверенные данные, а затем запускает:

```powershell
pwsh ./scripts/process-team-input.ps1 -InputPath <подготовленный-json> -Apply
```

Обработчик регистрирует источники, сохраняет и индексирует вложения, закрывает подтверждённые вопросы, перестраивает производные документы и записывает отчёт. Изменения публикуются отдельным запросом на изменение; объединение остаётся осознанным решением руководителя. Подробный путь команды описан в [`TEAM-INPUT.md`](./TEAM-INPUT.md), машинная политика — в [`TEAM-INPUT.json`](./TEAM-INPUT.json).

При первоначальном запуске [`scripts/setup-project.ps1`](./scripts/setup-project.ps1) автоматически создаёт для `main` GitHub Ruleset **«Проектная основа: единая версия»**. Поэтому руководителю не нужно вручную искать ещё не появившуюся проверку в интерфейсе GitHub. Для операции используются текущая авторизация `gh` и административные права пользователя; токен в проект не записывается.

## Четыре обязательных предохранителя

### 1. Отдельное рабочее место

Каждая нейросеть использует отдельный клон или отдельное рабочее дерево Git. Одновременная запись нескольких инструментов в одну папку запрещена: локальная блокировка не может надёжно различить независимые процессы и облачных исполнителей.

### 2. Паспорт изменения

До первого содержательного изменения нейросеть создаёт паспорт командой:

```powershell
pwsh ./scripts/start-ai-work.ps1 -Agent "Codex" -Task "Уточнить риски поставки" -Scope "CONTROLS.md","docs/03-delivery.md","HANDOFF.md"
```

Сценарий создаёт отдельную ветку при необходимости и фиксирует:

- уникальный номер работы;
- исполнителя и задачу;
- точную исходную редакцию канонической ветки;
- допустимые файлы и папки;
- номер интеграционной последовательности.

Пути к папкам заканчиваются `/`, например `docs/`. Символы `*`, абсолютные пути и выход через `..` запрещены.

### 3. Свежая основа и заявленные границы

Перед публикацией нейросеть выполняет:

```powershell
pwsh ./scripts/check-ai-coordination.ps1 -BaseRef origin/main
```

Проверка отклоняет работу, если она:

- начата не от текущей канонической редакции;
- выполнена непосредственно в `main`;
- не имеет ровно одного паспорта;
- затрагивает файл за пределами заявленной области;
- пытается пропустить общий интеграционный счётчик.

Та же проверка запускается в GitHub для каждого запроса изменений из ветки нейросети.

Ruleset требует эту проверку по имени заранее, ещё до первого запроса изменений. Если GitHub не был доступен во время первоначальной настройки, безопасно повторите операцию:

```powershell
pwsh ./scripts/configure-github-protection.ps1 -Apply
```

Сценарий можно запускать повторно: он обновляет собственный Ruleset вместо создания конфликтующих копий.

### 4. Последовательное объединение

Файл [`AI-INTEGRATION-STATE.json`](./AI-INTEGRATION-STATE.json) — технический турникет. Каждая нейросеть изменяет одну и ту же запись через сценарий запуска. Поэтому после объединения одной работы остальные ветки намеренно становятся устаревшими или конфликтующими и не могут незаметно образовать параллельную версию проекта.

Объединяется только один запрос изменений за раз. Следующий проходит повторную синхронизацию и все проверки уже с учётом предыдущего результата.

## Как синхронизировать оставшуюся работу

После изменения `main` нейросеть должна:

1. получить свежую каноническую ветку;
2. перенести свою ветку поверх неё и осмысленно разрешить содержательные конфликты;
3. перечитать [`HANDOFF.md`](./HANDOFF.md), [`STATUS.md`](./STATUS.md) и затронутые канонические файлы;
4. обновить паспорт и интеграционный счётчик:

```powershell
pwsh ./scripts/sync-ai-work.ps1 -ChangeId "номер-из-паспорта" -BaseRef origin/main
```

5. повторно собрать производные материалы и запустить проверки.

Сценарий синхронизации запускается только после завершённого переноса ветки. Он не разрешает содержательные конфликты вместо исполнителя.

## Как делить параллельную работу

Хорошее разделение — по независимому результату и каноническим файлам:

- одна нейросеть анализирует риски и меняет [`CONTROLS.md`](./CONTROLS.md);
- другая уточняет ценность и меняет [`OUTCOMES.md`](./OUTCOMES.md);
- третья проводит независимый разбор без записи, затем передаёт выводы интегрирующей нейросети.

Плохое разделение — поручить нескольким нейросетям одновременно переписать [`PROJECT-BRIEF.md`](./PROJECT-BRIEF.md), [`HANDOFF.md`](./HANDOFF.md) или один и тот же раздел. Пересечение областей не запрещено технически, но должно быть известно руководителю и почти всегда требует последовательного выполнения.

Файлы [`STATUS.md`](./STATUS.md) и [`PROJECT.md`](./PROJECT.md) являются производными. Их пересобирают после сведения содержательных изменений, а не используют как независимые источники правды.

## Чатовые нейросети без доступа к репозиторию

Такая нейросеть может дать анализ или черновик, но не создаёт собственную «версию проекта». Её ответ считается входным материалом и передаётся одной нейросети-оператору, которая сверяет его с текущим репозиторием, источниками и решениями, а затем вносит изменения по этому протоколу.

## Если проверка остановила работу

| Сообщение | Что делать |
|---|---|
| Исходная редакция устарела | Обновить ветку от `main`, перечитать изменившийся контекст и выполнить синхронизацию паспорта |
| Файл вне заявленной области | Расширить область осознанно и объяснить это в запросе изменений либо убрать лишнее изменение |
| Изменён интеграционный счётчик | Не выбирать вариант конфликта вслепую; взять состояние из свежего `main`, затем запустить синхронизацию |
| Нет паспорта | Вернуться к свежей основе и зарегистрировать работу до продолжения |
| Два паспорта | Разделить несвязанные изменения на отдельные ветки и запросы |

Автоматическую проверку нельзя отключать ради ускорения. Исключение требует явного решения владельца, описания риска и ручной проверки полного набора изменений.
