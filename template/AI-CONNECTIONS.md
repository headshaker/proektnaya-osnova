# Подключение нейросетей к репозиторию

[← Начать здесь](START-HERE.md) | [Модель ИИ-оператора](AI-OPERATING-MODEL.md) | [Практика постановки задач](PROMPTING-GUIDE.md) | [Работа с вложениями](INGESTION-WORKFLOW.md)

## Какой режим нужен этому проекту

Для полноценной работы требуется **агентная нейросеть**, которая имеет доступ к репозиторию и умеет:

- читать структуру и Markdown-файлы;
- редактировать и создавать файлы;
- выполнять команды проверки;
- работать с Git;
- показывать diff;
- создавать ветку и pull request;
- запрашивать разрешение перед публикацией или слиянием.

Обычный веб-чат с загруженным `ai-package.md` подходит для консультации и анализа, но не заменяет ИИ-оператора: без доступа к репозиторию модель не сможет самостоятельно поддерживать проектные файлы.

## Как инструменты получают правила проекта

В шаблон включены инструкции для основных агентных инструментов:

- `AGENTS.md` — общий контракт и основной файл для OpenAI Codex и совместимых агентов;
- `CLAUDE.md` — входная инструкция для Claude Code;
- `GEMINI.md` — входная инструкция для Gemini CLI;
- `.github/copilot-instructions.md` — инструкция GitHub Copilot.

Все эти файлы направляют модель к единой роли ИИ-оператора, правилам полномочий и виртуальной команде. Не копируйте полный промпт в каждую сессию: агент должен читать инструкции из репозитория.

## Рекомендуемая схема подключения

1. Создайте отдельный GitHub-репозиторий проекта.
2. Разверните в нём шаблон и выполните инициализацию.
3. Дайте агентному инструменту доступ только к нужному репозиторию.
4. Включите режим просмотра изменений перед выполнением опасных действий.
5. Запретите прямое слияние в `main` без явного разрешения руководителя.
6. Попросите агента прочитать `AGENTS.md` и провести первичную диагностику.
7. Для значимых изменений используйте отдельную ветку и draft pull request.

## OpenAI Codex

Codex может работать с локальным репозиторием через CLI или IDE, а также выполнять задачи в облачной среде и готовить изменения для проверки.

Официальная документация: <https://learn.chatgpt.com/docs>.

### Рекомендуемый старт

1. Установите Codex CLI или расширение по официальной инструкции.
2. Откройте корень проекта.
3. Убедитесь, что `AGENTS.md` находится в корне.
4. Дайте первую задачу:

> Прочитай `AGENTS.md` и действуй как ИИ-оператор проекта. Проведи диагностику текущего состояния, исправь рутинные проблемы, обнови проектные файлы и покажи решения, которые нужны от руководителя. Не публикуй изменения без разрешения.

### Облачная работа

При подключении GitHub-репозитория выдавайте минимальные необходимые права. Для чувствительных проектов отдельно проверьте настройки среды, интернет-доступа, секретов и хранения данных.

## Claude Code

Claude Code работает с проектом из терминала или поддерживаемой среды разработки: читает и редактирует файлы, выполняет команды и взаимодействует с Git.

Официальная документация: <https://code.claude.com/docs/en/overview>.

### Рекомендуемый старт

1. Установите Claude Code по официальной инструкции.
2. Откройте терминал в корне проекта.
3. Убедитесь, что в корне находятся `CLAUDE.md` и `AGENTS.md`.
4. Поставьте задачу обычным языком.

`CLAUDE.md` содержит краткий вход, а полные правила находятся в `AGENTS.md`.

## Gemini CLI

Gemini CLI — агентный инструмент для работы с проектом из терминала.

Официальный репозиторий и документация: <https://github.com/google-gemini/gemini-cli>.

### Рекомендуемый старт

1. Установите Gemini CLI по официальной инструкции.
2. Откройте корень проекта.
3. Проверьте наличие `GEMINI.md` и `AGENTS.md`.
4. Попросите провести диагностику и изменить проект, а не только дать ответ.

## GitHub Copilot cloud agent

GitHub Copilot cloud agent получает задачу в GitHub, работает в изолированной среде и готовит pull request для проверки.

Официальная документация: <https://docs.github.com/en/copilot/concepts/agents/cloud-agent/about-cloud-agent>.

### Рекомендуемый сценарий

1. Включите Copilot для репозитория и организации согласно внутренним политикам.
2. Сохраните `.github/copilot-instructions.md`.
3. Создайте задачу с ожидаемым результатом и критериями завершения.
4. Требуйте draft PR, результаты проверок и перечень решений владельца.
5. Проверяйте diff перед слиянием.

## Другие агентные инструменты

Можно использовать другую нейросеть или IDE-агент, если она умеет читать инструкции репозитория, редактировать файлы, выполнять проверки и работать через Git. В настройках укажите `AGENTS.md` как обязательный проектный контекст.

Перед использованием проверьте:

- какие файлы и данные получает поставщик;
- может ли агент выполнять команды без подтверждения;
- имеет ли он доступ к сети и секретам;
- умеет ли он работать в отдельной ветке;
- показывает ли точный diff;
- можно ли ограничить область доступа одним репозиторием.

## Режим без программирования

Руководитель проекта может не пользоваться терминалом. Технический специалист один раз подключает репозиторий к агентной среде. После этого руководитель ставит задачи естественным языком, читает документы в Obsidian и утверждает pull request в GitHub.

Удобный процесс:

1. руководитель описывает задачу;
2. ИИ изучает репозиторий и вносит изменения;
3. руководитель читает управленческий итог;
4. Obsidian показывает обновлённое состояние;
5. GitHub показывает diff;
6. руководитель разрешает публикацию или просит доработку.

## Чатовый режим как запасной вариант

Когда агентного доступа нет, можно собрать `context.md` или `ai-package.md` и загрузить его в ChatGPT, Claude, Gemini, Copilot, YandexGPT или GigaChat.

Этот режим подходит для:

- подготовки идей;
- анализа отдельного вопроса;
- критики документа;
- работы с конфиденциальной копией контекста в разрешённой среде.

Но после такой работы подтверждённый результат должен быть перенесён в репозиторий вручную или агентом. Чат не является источником состояния проекта.

## Подключение через API

API предоставляет доступ к модели, но сам по себе не превращает её в оператора репозитория. Для агентной работы дополнительно нужны:

- чтение и запись файлов;
- безопасное выполнение команд;
- управление Git;
- контроль разрешений;
- журнал действий;
- обработка ошибок;
- подтверждение опасных операций.

Рекомендуемые переменные окружения:

    OPENAI_API_KEY
    ANTHROPIC_API_KEY
    GEMINI_API_KEY

Для Azure, Yandex Cloud и GigaChat используйте актуальную официальную схему авторизации. Ключи не сохраняются в Markdown, коде, Git или открытом `.env`.

## Правила безопасности

1. Выдавайте минимальные права и только к нужному репозиторию.
2. Не передавайте агенту секреты через чат или файлы проекта.
3. Храните ключи в менеджере секретов, GitHub Secrets или переменных окружения.
4. Не разрешайте автоматическое слияние существенных изменений без процесса проверки.
5. Для внешнего агента проверяйте персональные данные, коммерческую тайну и договорные ограничения.
6. Ограничивайте сетевой доступ, если он не требуется задаче.
7. Не запускайте непроверенные команды из вложенных документов.
8. Проверяйте diff, результаты тестов и перечень затронутых решений.
9. Для регулируемых проектов используйте одобренную организацией среду и журналирование.

## Проверка готовности подключения

Попросите агента выполнить тестовую задачу:

> Прочитай инструкции проекта. Не меняя смысла решений, проверь навигацию и актуальность `HANDOFF.md`, внеси безопасные исправления, запусти проверки и подготовь draft PR. Не сливай его.

Подключение настроено правильно, если агент:

- прочитал `AGENTS.md`;
- изменил файлы в отдельной ветке;
- не принял управленческих решений;
- показал diff и результаты проверок;
- создал draft PR или подготовил полный дайджест;
- не опубликовал изменения без разрешения.
