# Как развивать «Проектную основу»

## Базовое правило

Изменение должно улучшать воспроизводимость, ясность или управляемость проектов и не переносить в универсальный шаблон предметные решения одного конкретного проекта.

## Рабочий процесс

1. Описать проблему и пример реального использования.
2. Проверить, нельзя ли решить её существующим файлом или правилом.
3. Изменить минимально необходимый набор файлов в `template`.
4. Обновить корневые инструкции, если меняется пользовательский сценарий.
5. Выполнить `pwsh ./scripts/test-template.ps1`.
6. В pull request описать совместимость, миграцию и результат проверки.

## Требования к изменениям

- отправляя вклад, участник соглашается распространять его в составе проекта на условиях лицензии MIT;
- русский язык является основным языком документации;
- технические идентификаторы и темы GitHub могут использовать латиницу;
- новая обязательная сущность должна иметь понятную пользу и владельца жизненного цикла;
- сетевые сервисы и community plugins не становятся обязательными без отдельного решения;
- производные файлы не редактируются вручную;
- изменение формата реестра сопровождается планом миграции;
- новый файл связан с навигацией и не дублирует существующий канон.

## Сообщения коммитов

Рекомендуемый формат:

    feat: добавить новую возможность
    fix: исправить проверку ссылок
    docs: уточнить правила ведения решений
    refactor: упростить структуру шаблона
    test: расширить сквозную проверку

## Версионирование

До утверждения стабильной схемы используется семантический ориентир:

- `0.x` — структура и интерфейсы могут меняться;
- минорная версия — новая совместимая возможность;
- исправление — совместимое исправление ошибки;
- `1.0` — стабильный минимальный состав и документированный путь обновления.
